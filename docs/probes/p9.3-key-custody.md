# P9.3 — Per-employee provider-key custody (#103)

> Consumes **ADR-0009** (provider-key custody: Stronghold + OS-keychain fallback).
> Part of **P9 — BYO-AI Providers** (#80), L5 Cockpit program (#77).

## What shipped (firm-OS workspace, CI-verified)

Provider keys are **retrievable** secrets, so they cannot reuse `ApiKeyService`'s
hash-only store (ADR-0009 "hash-only gap"). P9.3 adds the custody **seam** and the
testable backend to the workspace, and the route surface to the API:

| Piece | Where | Notes |
|---|---|---|
| `ProviderKeyStore` trait | `kanbrick-providers/src/custody.rs` | object-safe, `Send + Sync`; every method takes the caller's `user_id` |
| `KeyId` / `KeyMetadata` / `KeyStoreError` | same | `KeyMetadata` has **no secret field** — the only shape that crosses outward |
| `InMemoryKeyStore` | same | per-user map behind a `Mutex`; the reference for the isolation invariant; the no-`libsodium` fallback |
| `POST/GET/DELETE /me/provider-keys` | `kanbrick-api/src/provider_keys.rs` | `require_clearance` + `AuditLog`, metadata-only reads, namespaced by `FirmContext.user_id` |
| `AppState.provider_keys` + `with_provider_keys()` | `kanbrick-api/src/lib.rs` | defaults to in-memory; the cockpit injects the Stronghold backend |

The **isolation invariant** is structural, not a check to remember: the store's
outer map is keyed on `user_id`, and every route reads `ctx.user_id` from the
host-authoritative JWT (never the request path/body), so a caller can only ever
touch their own namespace. The integration test (`tests/provider_keys.rs`) proves
a second user sees none of the first's keys and a cross-user `DELETE` is a `404`,
plus 401-when-unauthenticated, metadata-only reads, and one audit entry per action.

## The durable backends (cockpit build) — ADR-0009 §4

ADR-0009 adopts **IOTA Stronghold** as the primary at-rest enclave, with the **OS
keychain / Tauri secure store** as the documented fallback where `libsodium` is
unavailable. Both are concrete `ProviderKeyStore` implementations injected via
`AppState::with_provider_keys(...)`, so the routes, the audit, the clearance gate,
and the namespacing above are **identical** regardless of backend — only where the
ciphertext rests differs.

These backends live on the **cockpit side** (`cockpit/src-tauri`), not in this
workspace, because ADR-0009 records that Stronghold's one native `-sys` dependency
(`libsodium-sys-stable`, + `cc`/`pkg-config`) must enter the **Cockpit build, not
the firm-OS workspace** — the cockpit CI runner provides the C toolchain; the
firm-OS `cargo build --workspace` stays free of it (and of Stronghold's 179-crate
closure). The `auth::Session` seam (ADR-0009 §3) holds a `dyn ProviderKeyStore`,
so wiring the Stronghold backend in is an injection, not a refactor.

### Stronghold backend sketch (cockpit)

```text
StrongholdKeyStore {
    snapshot_dir,            // per-workstation, on durable storage
    unlock: HostDerivedKey, // never crosses IPC; derived host-side
}
impl ProviderKeyStore for StrongholdKeyStore {
    // vault/client path namespaced by user_id (Uuid) -> a record per KeyId
    // put     -> write record (provider, label, created_at, secret) to the user's vault, persist snapshot
    // get_secret/list/delete -> open the user's vault only; a wrong user_id opens a different (empty) namespace
}
```

A compromise of one user's unlock material cannot read another user's vault
(ADR-0009 §2): the unlock key is host-derived and the vault path is keyed on
`user_id`.

## Honest environment note

The live Stronghold **store → read round-trip** is **not exercised here**, for the
same reasons recorded in ADR-0009 / `p8.2-stronghold-spike.md`: crate *tarball*
downloads are proxy-blocked (HTTP 403) and the backend needs `libsodium` at build
time. The round-trip and the cross-user-isolation test against the *real* enclave
are first run on a network-capable machine / the cockpit CI. What is proven here
is the **seam, the namespacing model, the route/audit/clearance wiring, and the
isolation invariant** — all against the in-memory backend, which CI compiles and
tests. Swapping in Stronghold changes the at-rest bytes, not the contract.
