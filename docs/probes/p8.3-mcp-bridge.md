# Probe ‚Äî MCP bridge for loop tool-calls (P8.3)

> Probe note ‚Äî P8.3 (#95) ¬∑ part of P8 ‚Äî Upstream De-Risk (#79).

## Context

Loops (P11) will call external tools. Tachyon-Mesh ships an **MCP (Model Context
Protocol) server** (`tachyon-mcp`, over stdio). This probe evaluates wrapping it as
a Kanbrick host service so a loop step can invoke MCP tools under
**host-authoritative identity** ‚Äî **without** adopting the Tachyon runtime
(ADR-0014).

## Recommendation: managed sidecar, not a HostServices backend

Run `tachyon-mcp` as a **managed sidecar**, reusing the **Phase-7.2
`SidecarSupervisor`** pattern (spawn ‚Üí `/health`-gate ‚Üí supervise ‚Üí kill), rather
than wrapping it behind the `HostServices` trait.

Rationale: `HostServices` (`kanbrick-mesh/src/services.rs`) is the **guest‚Üîhost
graph / event ABI** (`query_graph` / `emit_event`) ‚Äî clearance-filtered graph
access for WASM guests. MCP tool-calls are a **different concern**: external tools,
not graph reads. A supervised sidecar keeps MCP **isolated, restartable, and out of
the wasm host trait**, and composes with ADR-0014 (no second runtime, no
`core-host` coupling).

## Identity passthrough sketch

A loop step's MCP call runs under the caller's **authorized `ProjectScope`**:

1. The host mints a per-invocation capability (`InvocationCaps::mint`,
   `kanbrick-api/src/caps.rs`) bound to the caller's `FirmContext`.
2. The MCP sidecar receives **only** the opaque capability plus the specific tool +
   arguments the `ProjectScope` authorizes ‚Äî never the identity bytes.
3. Results return to the host, which **re-authorizes** before applying them.

This mirrors ADR-0008's capability model: identity stays host-side; only the
minimum crosses the boundary. The `x-kanbrick-internal-token` (fail-closed) gates
the host‚ÜîMCP-sidecar control channel.

## What we could verify here

Standing up `tachyon-mcp` + a live tool-call requires the `tachyon-mesh` submodule,
which is proxy-blocked (HTTP 403) this session (same constraint as P8.1). The
recommendation is grounded in the documented `tachyon-mcp` shape plus the
**existing, in-repo** sidecar (P7.2) and capability (ADR-0008) patterns; the live
tool-call is deferred to a submodule-capable machine. **No coupling to `core-host`
is introduced.**
