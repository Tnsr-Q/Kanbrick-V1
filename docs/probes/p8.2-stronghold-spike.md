# Probe ‚Äî IOTA Stronghold as the per-employee secret enclave (P8.2)

> Probe note ‚Äî P8.2 (#94) ¬∑ part of P8 ‚Äî Upstream De-Risk (#79). Backs ADR-0009.

## What we set out to do

Decide whether IOTA Stronghold can be the per-employee provider-key enclave for
BYO-AI (P9) as a **minimal-dependency standalone library** (no Tachyon-Mesh
`core-host`, per ADR-0014), and round-trip one secret through it. The contrast is
the existing `ApiKeyService` (`kanbrick-auth/src/apikey.rs`), which stores
**hashes only** (SHA-256 via `query_hash`, `constant_time_eq`) ‚Äî it validates a
presented secret but can never *return* one (the "hash-only gap").

## Dependency-closure evidence

`cargo add iota_stronghold` against the crates.io **index** (reachable through the
agent proxy) resolved cleanly:

| --- | --- |
| Version | `iota_stronghold` v2.1.0 |
| Transitive crates | **179**, all "Rust 1.94.1 compatible" ‚Üí **zero toolchain bump** (matches our pin) |
| Pulls `core-host`? | **No** ‚Äî satisfies ADR-0014 (bounded primitive, not the runtime) |

## What could and could not be verified here

- ‚úÖ The **dependency closure** resolves on our pinned toolchain (1.94.1) with no
  `core-host` and no second async runtime ‚Äî Stronghold is a genuine standalone,
  minimal-dependency enclave.
- ‚è≥ The **live store ‚Üí read round-trip could not be executed** in this
  environment: crate *tarball* downloads are blocked by the agent proxy (HTTP
  403); only the index resolves, so the source cannot be compiled here. The
  round-trip is deferred to a network-capable machine / the cockpit CI (which also
  provides `libsodium` for the `-sys` dep).

## Namespacing & cross-user isolation sketch

- One Stronghold vault / snapshot **per employee, keyed by `FirmContext.user_id`**
  (the JWT-derived `Uuid`). Reads / writes happen **host-side only**; the webview
  never sees plaintext, consistent with the host-authoritative identity of
  ADR-0016.
- The unlock key is **host-derived per user**; a compromise of one user's unlock
  material cannot read another user's vault (distinct vault paths under distinct
  unlock keys).
- The P7.3 in-memory `auth::Session` is the **seam**: durable Stronghold custody
  slots in behind the same interface without changing callers.

## Conclusion

Adopt Stronghold as the primary enclave (‚Üí **ADR-0009**), with the OS keychain /
Tauri secure store kept as a documented **fallback** for environments where the
`libsodium-sys-stable` native dependency is unavailable. The round-trip + the
cross-user isolation test are first exercised on a network-capable machine / CI.
